- [ADR-024] V2.2 passive inputs / wearables (roadmap §B): objective recovery data (Whoop + Apple Health) flows into the observation spine as a NEW raw/import provenance path — 5 metrics (recovery_score/hrv_rmssd/hrv_sdnn/resting_hr/sleep_device, domain 'recovery') behind two new tables (device_connections one row per subject+provider holding OAuth tokens; import_batches append-only audit rows) and one atomic RPC `import_observations` (mirrors save_daily_checkin — inserts the batch row server-side, then upserts observations hardcoded to source='import'; since `source` is part of the dedupe key the upsert can only converge import rows onto import rows, never onto manual/emitted — no delete phase, the upsert IS the edit). HRV SPLIT into two keys (hrv_rmssd Whoop-only, hrv_sdnn Apple-only) — RMSSD and SDNN are different measurands (~1.5-2x apart), a shared series would corrupt personal z-baselines at device switchover; resting_hr stays ONE shared key (same measurand, few-bpm cross-device offset accepted, baseline re-forms in ~3 weeks); sleep_device is its OWN key, never folded into sleep_duration (device ≠ subjective report, and the engine filters by metric_key only — mixing would silently pollute baselines). Apple bridge = file import (Health Auto Export JSON via a Server Action, zero new server secrets on that path) since HAE automations only run with the phone unlocked, ruling out REST push; Whoop = full OAuth2 connector (client.ts token exchange + rotation-aware refresh, map.ts wake-date attribution via the linked sleep's `end` skipping naps/non-SCORED, sync.ts conditional-UPDATE claim so only one caller syncs + refresh-persists-before-pull ordering + 30d first-sync/7d-overlap window) built NOW but env-gated behind `WHOOP_CLIENT_ID/SECRET/REDIRECT_URI` (null config -> UI shows "Próximamente") since Whoop devices are weeks away — validated with synthetic fixtures, real OAuth is a device-arrival ritual (see skills.md). Device sleep PREFILLS the check-in's sleep_hours field (editable) rather than replacing it outright — manual entry remains THE engine series, matching ADR-012's subjective-readiness philosophy. First route handler in the repo (/api/whoop/callback), still behind the proxy auth gate (user must already be logged in to connect). NO new flags/señales/patterns from device data (out of scope, V2.3 territory) — rejected: single shared HRV key (baseline corruption at device switchover), Whoop webhooks (unauthenticated ingest breaks the no-service-role rule, pull-only instead), replacing manual sleep entirely (subjective readiness is deliberately part of the philosophy, ADR-012), native HealthKit/direct REST push (unlocked-phone limitation + per-user ingest tokens the file path already avoids) (2026-07-03)
